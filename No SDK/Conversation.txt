The key distinction is:

Spotify catalog endpoints do not require user-specific scopes, but they still require an access token.

So /artists/{id}, /artists/{id}/top-tracks, and /search are not “public unauthenticated” endpoints. They can be used without permission to read your playlists/profile/library, but Spotify still requires OAuth authentication for the request.

Because this app is currently static browser-only, we cannot safely use Spotify’s Client Credentials flow. That flow needs a client_secret, and putting that secret in app.js would expose it to anyone who opens DevTools.

So the Connect Spotify button exists to get a browser-safe OAuth PKCE access token using only your public Client ID.

Options:

Keep current browser-only app

Needs Connect Spotify
No client secret exposed
No Spotify SDK
Remove the button entirely

Add a tiny backend/serverless function
Backend uses Client ID + Client Secret with Client Credentials
Browser calls your backend, backend calls Spotify
Rename the button

“Authorize Spotify API” or “Load Spotify Data” may be less misleading than “Connect Spotify”
So you were right that these are not really “login to my personal Spotify account” features. But Spotify still requires an access token, and PKCE is the safe way to get one from a static browser app.

Yes, but not safely in a purely browser-only app.

Your Client ID is safe to put in frontend code. Your Client Secret is not. If we put the secret in app.js, anyone can view it in the browser and use your Spotify app credentials.

The safe setup is:

Browser app -> tiny local backend -> Spotify API

The backend keeps the Client Secret server-side, uses Spotify’s Client Credentials flow to get an app access token, then proxies requests for tracks/artists.

For this demo, we can absolutely do that safely if you’re running it locally. I’d add something like:

server.js
.env with:
SPOTIFY_CLIENT_ID=...
SPOTIFY_CLIENT_SECRET=...
frontend calls /api/search, /api/artists, /api/artists/:id/top-tracks
no Spotify login button needed
One important caveat: if this is deployed publicly later, you’ll want rate limiting and origin checks so people cannot freely use your backend as a Spotify proxy.

I can convert this app to that backend-based version next.
